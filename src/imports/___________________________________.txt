Обзор различных методов адаптивной верстки:
Медиазапросы (media queries): Основной инструмент для создания адаптивных дизайнов, с помощью которого можно изменять стили элементов в зависимости от характеристик устройства, таких как ширина экрана, разрешение и ориентация.

Flexbox: Современная технология для создания гибких и отзывчивых макетов, особенно полезна для выравнивания элементов по вертикали и горизонтали.
CSS Grid: Сложная и мощная система для создания адаптивных макетов с сеткой.

Для успешного выполнения работы студент должен знать:

Основы архитектуры «клиент-сервер».

Модель OSI/TCP-IP, роль протоколов HTTP/HTTPS.

Структуру HTTP-сообщений (запросы и ответы).

Основные методы HTTP-запросов (GET, POST и др.).

Классы и основные коды состояния HTTP (1xx, 2xx, 3xx, 4xx, 5xx).

Назначение ключевых HTTP-заголовков (User-Agent, Content-Type, Cookie, Cache-Control и др.).

4. ПОРЯДОК ВЫПОЛНЕНИЯ РАБОТЫ (Общий для всех вариантов)
Часть 1. Подготовка и общее исследование

Откройте инструменты разработчика.

Запустите браузер Google Chrome или Mozilla Firefox.

Откройте панель разработчика с помощью клавиши F12 или комбинации Ctrl+Shift+I (Cmd+Opt+I на Mac).

Перейдите на вкладку Network («Сеть»).

Настройте панель Network.

Убедитесь, что кнопка записи («Record network log») активна (обычно красного цвета).

Поставьте галочку «Preserve log» («Сохранять журнал»), чтобы записи не очищались при переходе между страницами.

Очистите текущий журнал, нажав на кнопку «Clear» («Очистить»).

Зафиксируйте сетевой трафик.

В адресной строке браузера введите URL одного из сайтов, указанных в вашем индивидуальном задании (если необходимо, включите VPN!), и нажмите Enter.

Наблюдайте, как в панели Network заполняется список всех сетевых запросов, которые браузер выполняет для загрузки страницы и сопутствующих ресурсов (HTML, CSS, JS, изображения, шрифты, API-вызовы).

Проведите общий анализ.

Обратите внимание на столбцы: Name (имя ресурса), Status (код состояния), Type (тип ресурса), Initiator (инициатор запроса), Size (размер), Time (время загрузки).

Найдите самый первый запрос в списке (обычно это документ с типом document). Его статус должен быть 200 OK (или 304 Not Modified).

Часть 2. Детальный анализ запроса и ответа

Выберите целевой запрос. Выберите из списка любой запрос, например, загрузку основного HTML-документа или запрос к API.

Исследуйте вкладку «Headers».

General Headers: Изучите общую информацию: Request URL (адрес), Request Method (метод, например, GET), Status Code (код состояния).

Response Headers: Проанализируйте заголовки, которые сервер отправил браузеру. Найдите и запишите в отчёт назначение 3-5 заголовков (например, Content-Type, Cache-Control, Set-Cookie).

Request Headers: Изучите заголовки, которые браузер отправил серверу. Найдите и запишите в отчёт назначение 3-5 заголовков (например, User-Agent, Accept, Cookie).

Исследуйте другие вкладки (опционально, по требованию варианта).

Preview / Response: Позволяет увидеть обработанный ответ от сервера (например, отрендеренную HTML-страницу или JSON-данные).

Timing: Показывает детальную временную диаграмму выполнения запроса (DNS-поиск, установка соединения, SSL-рукопожатие, ожидание ответа сервера и т.д.).

Это web‑сервис для формального моделирования систем и процессов, где пользователь пишет код диаграммы в специальном языке, а система превращает его в визуальную диаграмму, хранит, версионирует и умеет по тексту или картинке восстановить код диаграммы.

1. Общая идея и роль системы
Система решает задачу «диаграммы как код» для строгих нотаций (UML, BPMN, сети Петри, ERD и др.):

Пользователь описывает структуру или процесс в текстовом DSL (твоём языке).
Сервис преобразует текст в формальную внутреннюю модель.
На основе модели рендерит диаграмму (SVG/PNG/HTML) и даёт её редактировать через изменение кода.
Диаграммы можно сохранять, скачивать, просматривать историю версий и входных запросов.
Отдельный модуль умеет по описанию на естественном языке (и позже по изображению) построить черновой код диаграммы.
Система может использоваться как учебный и инженерный инструмент: для курсовых, для проектирования бизнес‑процессов, архитектур, протоколов, формального анализа.

2. Основные пользовательские сценарии
2.1. Ручное создание диаграммы (Diagram as Code)
Пользователь открывает web‑страницу сервиса.
В левом окне (консоль/редактор) пишет код диаграммы на DSL, например:
выбирает тип: diagram bpmn "Заказ товара" { ... },
описывает элементы: задачи, события, шлюзы, исполнителей,
задаёт связи между ними.
Нажимает «Сгенерировать» (или превью обновляется автоматически).
В правом окне видит визуальную диаграмму, построенную по коду.
При изменении кода диаграмма пересчитывается (парсится заново, валидируется и перерисовывается).
Пользователь может:
сохранить диаграмму в аккаунт,
скачать её как PNG/SVG/JSON,
экспортировать код в другие форматы (например, DOT, PlantUML/Mermaid‑подобный синтаксис, BPMN XML),
вернуть одну из предыдущих версий.
2.2. Генерация диаграммы по тексту
Пользователь выбирает режим «Текст → диаграмма».
Вводит описание: «Опиши процесс оформления заказа: клиент делает заказ, система проверяет оплату…».
Указывает желаемую нотацию: BPMN, UML Activity, сеть Петри.
Система:
анализирует текст с помощью модели,
строит внутреннее представление процесса (список шагов, ветвления, условия),
генерирует DSL‑код (примерный, черновой),
отображает получившийся код в редакторе и сразу же визуальную диаграмму.
Пользователь руками дорабатывает код и сохраняет.
2.3. Генерация по картинке (диаграмма → код)
Пользователь загружает картинку с нарисованной диаграммой (скрин, фото с доски, рисунок).
Выбирает нотацию, если её можно определить.
Система:
распознаёт фигуры (прямоугольники, ромбы, стрелки, подписи),
строит граф элементов и связей,
сопоставляет распознанные элементы с понятиями выбранной нотации (task, event, gateway, place/transition и т.п.),
генерирует DSL‑код, максимально близкий к исходной диаграмме.
Пользователь видит код и полученную диаграмму, редактирует и сохраняет.
2.4. Работа с историей и версиями
Каждое сохранение создаёт новую версию: сохраняется код DSL, метаданные (тип диаграммы, нотация, пользователь, время).
Можно открыть список версий, сравнить (diff текста, основные изменения в структуре), откатиться к любой версии.
Для AI‑сценариев отдельно хранится «история запросов»:
текстовый запрос пользователя,
ответ системы (сгенерированный DSL),
дата/время,
тип запроса (текст/картинка/гибрид).
Это превращает сервис в «Git для диаграмм», но с фокусом на формальные нотации.

3. Внутренняя архитектура (крупными блоками)
Логически система делится на несколько уровней:

Web‑интерфейс (frontend).
REST/GraphQL‑backend.
Ядро моделирования (DSL → модель → layout → рендер).
Модуль хранения и версий.
AI‑модуль (NLP/CV → модель).
Модули импорт/экспорт стандартных форматов.
3.1. Внутренняя модель (ядро)
Внутри всё сводится к общей формальной модели: ориентированный граф с типизированными узлами и рёбрами.

Узел:
id,
тип (task, event, gateway, place, transition, entity, process и т.п.),
атрибуты (имя, текст, роль, временные характеристики, цвет, позиция, иконка).
Ребро (связь):
id,
тип (поток управления, поток данных, ассоциация, сообщение),
откуда/куда (source, target),
атрибуты (условие, метка).
Диаграмма/модель:
набор узлов и рёбер,
информация о нотации,
layout‑метаданные (координаты, уровни).
Для разных нотаций задаются «профили»:

профиль BPMN: перечисление допустимых типов элементов (StartEvent, Task, ExclusiveGateway…), разрешённых связей (SequenceFlow, MessageFlow) и ограничений;
профиль UML Activity: действия, начальные/конечные точки, решения, слияния, потоки;
профиль Petri: только places, transitions, arcs, с целочисленной маркировкой.
Все диаграммы любых нотаций проходят через одно и то же «ядро графа»; различаются только наборы типовых элементов и правила валидации.

3.2. Язык (DSL) и парсер
Твой DSL — текстовый язык, который описывает:

декларацию диаграммы (тип нотации, название, настройки);
список элементов (узлов) с их свойствами;
связи между элементами;
дополнительные параметры (стили, группы, swimlane, комментарии).
Парсер:

Получает текст от пользователя.
Выполняет лексический анализ (разбиение на токены: ключевые слова, идентификаторы, литералы).
Выполняет синтаксический анализ по грамматике DSL, строит синтаксическое дерево.
Преобразует дерево в внутреннюю модель (узлы, рёбра, профили нотаций).
Передаёт модель в валидатор.
Валидатор:

проверяет соответствие нотации: например, в BPMN стартовое событие не может иметь входящих последовательных потоков, в сети Петри переходы не соединяются напрямую, а только через места;
выдаёт сообщения об ошибках с указанием строк/символов и возможных исправлений;
может автоматически исправлять некоторые ошибки (добавить EndEvent, если нет, предложить тип элемента и т.п.).
3.3. Layout и рендер
Layout:

получает на вход внутренний граф;
рассчитывает координаты узлов и маршруты рёбер (расположение в слоях, обход пересечений);
учитывает ограничения нотации: swimlane в BPMN, декомпозицию в IDEF0, уровни DFD, структуру ER‑диаграммы.
Рендер:

берёт разложенный по координатам граф;
рисует фигуры (прямоугольники, окружности, ромбы, стрелки, подписи) в формате SVG/Canvas/HTML;
поддерживает стили (цвета, шрифты, иконки, темы).
В интерфейсе это отображается:

как интерактивное SVG (с zoom, pan, подсветкой);
при необходимости как PNG для скачивания.
4. Backend и хранение
4.1. REST/GraphQL‑API
Backend предоставляет набор API для frontend и внешних клиентов:

аутентификация (регистрация, логин, токены);
управление диаграммами:
создать диаграмму (тип нотации, пустой или с шаблоном),
получить текущую версию,
сохранить новую версию (отправить DSL, получить id версии),
удалить диаграмму или версию;
генерация изображений:
POST DSL → SVG/PNG,
POST DSL → JSON‑модель;
AI‑сервисы:
POST описание текста → DSL → диаграмма,
POST картинка → DSL → диаграмма;
импорт/экспорт:
POST BPMN XML / DOT / PlantUML / Mermaid → DSL,
GET диаграммы в этих форматах.
4.2. Схема данных
В БД хранится минимум:

Пользователи:
id, логин, хеш пароля, настройки.
Диаграммы:
id, владелец, название, тип нотации, текущая версия, теги, дата создания/обновления.
Версии диаграмм:
id, diagram_id,
DSL‑код,
краткие метаданные (количество узлов/рёбер, валидность, размер),
создано кем (user/AI),
дата и, возможно, комментарий.
История запросов:
id, diagram_id (или null, если новая),
тип (text→DSL, image→DSL),
текст запроса / ссылка на картинку,
ссылка на сгенерированную версию,
ответ (успех/ошибка).
Это позволяет:

быстро восстановить любую версию диаграммы;
анализировать, как AI‑модуль влияет на результат;
строить diff’ы по тексту DSL и по структуре диаграммы.
5. AI‑модуль и работа с естественным языком/изображениями
5.1. Текст → диаграмма
Модуль работает в несколько шагов:

Нормализация запроса: удаление лишнего текста, выделение ключевых сущностей и действий.
Семантический разбор:
извлекаются участники процесса, шаги, условия, варианты ветвления, данные;
строится промежуточное представление в виде JSON‑графа: список узлов (тип, роль) и связей (последовательность, условные переходы).
Маппинг в конкретную нотацию:
если выбрана BPMN: узлы преобразуются в Task, Event, Gateway и т.п.;
если UML Activity: в действия, решение, слияние;
если сеть Петри: в places и transitions.
Генерация DSL:
JSON‑модель сериализуется в твой язык по правилам: объявление диаграммы, узлов, связей;
результат сразу отправляется в парсер/валидатор и рендерится.
Обратная связь пользователю:
пользователю показывается сгенерированный DSL, подсвечиваются места, которые AI пометил как «сомнительные» (например, неясные условия);
пользователь исправляет и сохраняет.
5.2. Картинка → диаграмма
Этапы:

Обработка изображения:
детекция форм (фигур, линий, текстовых полей),
распознавание текста внутри фигур (OCR),
определение связей между фигурами по стрелкам и линиям.
Построение графа:
каждая фигура становится кандидатом в узел,
каждая стрелка — кандидатом в ребро,
текст поверх/внутри узла используется как имя/тип.
Классификация в терминах нотации:
по форме и ключевым словам классифицируется тип элемента (например, круг с «Start» → StartEvent BPMN, ромб → gateway),
строится модель выбранной нотации (или выбирается нотация автоматически по типам элементов).
Генерация DSL и визуализация, как и в текстовом сценарии.
6. Что отличает систему от существующих инструментов
Единый DSL, но поддержка нескольких формальных нотаций (UML, BPMN, сети Петри, IDEF, DFD, ERD и др.).
Не просто «рисовалка», а «формальный редактор»: валидатор строго проверяет правила нотаций.
Встроенная история и трассировка: хранение не только диаграмм, но и запросов/описаний, из которых они получены.
Двусторонний мост между человеком и машиной:
человек пишет код → формальная модель → красивое изображение;
человек даёт текст/картинку → система строит код, который можно анализировать, версионировать, интегрировать с другими инструментами.
Если хочешь, могу:

оформить это как раздел «Общая характеристика системы» + «Назначение и функции» для РПЗ,
или сделать ещё более формальное описание в стиле требований (ИР, НФР, UML‑диаграммы самого сервиса, описанные в твоём DSL по шагам). Ниже — максимально развернутые требования и полный набор use case’ов с точки зрения курсового/реальной системы «диаграммы как код». тип диаграммы (процессная, структурная и т.п.).geeksforgeeks+2
Позволять открывать существующие диаграммы из списка пользователя.

разобрать код DSL (лексический и синтаксический анализ);

отображать список ошибок с указанием:

типа ошибки;
пояснения и, при возможности, рекомендации по исправлению.
Поддерживать интерактивный режим:

Система должна:

Обеспечивать выбор нотации при создании диаграммы и фиксировать её в метаданных.
При изменении нотации (если разрешено) выполнять попытку трансформации модели с уведомлением о потерях/несоответствиях.
1.5. Валидация и диагностика
Система должна:

Выполнять структурную валидацию диаграммы:
отсутсвие «висящих» узлов без входящих/исходящих связей (с учётом нотации);
наличие начальных/конечных элементов там, где это требуется (Start/End Event в BPMN, Initial/Final Node в UML Activity и т.п.);
соответствие типов элементов и связей требованиям нотации.modeling-languages+1
Показывать предупреждения (warnings) для нежелательных, но формально допустимых конструкций (например, слишком много элементов в одном слое, трудно читаемая диаграмма).
При необходимости давать рекомендации:
разнести диаграмму на подпроцессы;
использовать события вместо условий на стрелках и т.д.
1.6. Визуализация диаграмм

сделать выбранную версию текущей;

2. Нефункциональные требования
2.1. Производительность

Обеспечивать рендер типичной диаграммы (до 100–200 узлов, 300–400 связей) не более чем за 1–2 секунды при средних нагрузках.ibm+1
Поддерживать интерактивную работу редактора:
задержка между изменением кода и обновлением превью — не более 0.5–1 секунды при небольших диаграммах (до 50 узлов).
Масштабироваться по нагрузке:
возможность вынести рендеринг и AI‑модуль на отдельные сервисы;
возможность асинхронной очереди запросов (особенно для AI и тяжёлого рендеринга).
2.2. Надёжность и отказоустойчивость
Система должна:

Гарантировать сохранность диаграмм и версий:
транзакционное сохранение;
регулярные резервные копии БД.
Корректно обрабатывать ошибки:
ошибки парсинга, AI, рендеринга — не приводят к потере данных;
пользователь получает понятное сообщение об ошибке.
Работать в режиме с доступностью не ниже заданного SLA (например, 99% для учебного сервера).modeling-languages+1
2.3. Масштабируемость
Система должна:

Позволять горизонтально масштабировать:
frontend (через CDN/статический хостинг);
backend (через балансировщик и несколько инстансов).
Разделять нагрузку:
отдельный процесс/сервис для рендеринга;
отдельный сервис для AI‑генерации диаграмм.
2.4. Безопасность
Система должна:

Хранить пароли в зашифрованном виде (хеширование с солью).
Обеспечивать защиту API (JWT/HTTPS).
Ограничивать доступ к диаграммам по правам пользователя.
Защищать от типичных web‑угроз (XSS, CSRF, SQL‑injection и т.п.).
Обеспечивать контроль лимитов для AI‑запросов:
ограничение по частоте;
защита от чрезмерно больших запросов/изображений.
2.5. Юзабилити
Система должна:

Иметь простой, интуитивно понятный интерфейс:
разделение на «код» и «диаграмму»;
понятные кнопки: Создать, Сохранить, Экспорт, История, AI‑генерация.
Предоставлять:
встроенные примеры диаграмм для каждой нотации;
справку по синтаксису DSL (cheatsheet).lucidchart+1
Обеспечивать доступность:
корректное отображение на распространённых разрешениях экранов;
поддержку клавиатурных сочетаний (Ctrl+S, Ctrl+Z и т.п.).
2.6. Расширяемость и модульность
Система должна:

Быть построена модульно:
отдельное ядро моделирования (DSL → модель);
отдельные модули нотаций (BPMN, UML, Petri и т.п.);
отдельный слой layout/рендеринга;
отдельный AI‑слой.utdallas+1
Позволять добавлять новые нотации без изменения существующего ядра (через интерфейсы профилей/плагинов).
Позволять заменять компоненты (например, движок layout) с минимальными изменениями.
2.7. Интероперабельность
Система должна:

Работать с внешними системами через понятные форматы:
JSON‑API;
DOT/BPMN XML/XMI;
экспорт в стандартные форматы диаграмм.daily+1

3. Use Cases (прецеденты)
Ниже — ключевые прецеденты системы. Для UML Use Case диаграммы актор обычно «Пользователь» и опционально «AI‑сервис».

3.1. UC1: Регистрация и вход пользователя
Актор: Пользователь.
Цель: Получить персональный доступ к своим диаграммам.
Основной сценарий:
Пользователь открывает сайт и выбирает «Регистрация».
Вводит логин, пароль, e‑mail.
Система проверяет корректность, создаёт учётную запись.
Пользователь входит в систему.
Альтернативы:
Ошибки валидации, занятый логин, неверный пароль.
3.2. UC2: Создание новой диаграммы
Актор: Пользователь.
Цель: Создать пустую диаграмму или на основе шаблона.
Основной сценарий:
Пользователь выбирает «Создать диаграмму».
Указывает название, нотацию, тип диаграммы.
Система создаёт запись и открывает редактор с шаблоном кода.
Пользователь видит пустую/типовую диаграмму в окне просмотра.
3.3. UC3: Редактирование диаграммы в DSL
Актор: Пользователь.
Цель: Описать/изменить диаграмму через текст.
Основной сценарий:
Пользователь вводит/редактирует код в консоли.
Нажимает «Сгенерировать» или ожидает авто‑обновления.
Система парсит код, валидирует, рендерит диаграмму.
Система отображает диаграмму в окне просмотра, ошибки — в панели ошибок.
Пользователь корректирует ошибки при необходимости.
Альтернативы:
Неправильный синтаксис — диаграмма не обновляется, показываются ошибки.
3.4. UC4: Просмотр диаграммы
Актор: Пользователь.
Цель: Просмотреть диаграмму и её структуру.
Основной сценарий:
Пользователь открывает диаграмму.
Система отображает текущую версию диаграммы.
Пользователь масштабирует, двигает, наводит курсор на элементы для подсказок.
3.5. UC5: Сохранение диаграммы и версии
Актор: Пользователь.
Цель: Сохранить текущее состояние диаграммы.
Основной сценарий:
Пользователь нажимает «Сохранить».
Система:
сохраняет текущий DSL как новую версию;
обновляет метаданные диаграммы.
Пользователь получает подтверждение.
3.6. UC6: Просмотр истории версий и откат
Актор: Пользователь.
Цель: Вернуться к предыдущему состоянию диаграммы.
Основной сценарий:
Пользователь открывает «Историю».
Видит список версий с датами, комментариями.
Выбирает две версии для сравнения.
Система показывает diff и визуальные отличия.
Пользователь выбирает версию и жмёт «Сделать текущей».
Система создаёт новую версию на основе выбранной (фактически откат).
3.7. UC7: Экспорт диаграммы
Актор: Пользователь.
Цель: Выгрузить диаграмму в файл (изображение или код).
Основной сценарий:
Пользователь выбирает «Экспорт».
Выбирает формат: SVG, PNG, DOT, BPMN XML, PlantUML/Mermaid и т.п.
Система выполняет трансформацию.
Пользователь скачивает файл.daily+1
3.8. UC8: Импорт диаграммы из внешнего формата
Актор: Пользователь.
Цель: Импортировать диаграмму из другого инструмента.
Основной сценарий:
Пользователь выбирает «Импорт».
Загружает файл (DOT, BPMN XML, XMI и др.).
Система парсит файл, строит внутреннюю модель.
Система генерирует DSL и создаёт новую диаграмму.
Пользователь может её редактировать.
3.9. UC9: Автоматическая генерация диаграммы по тексту
Актор: Пользователь, AI‑сервис.
Цель: Быстро получить черновую диаграмму из текстового описания.
Основной сценарий:
Пользователь открывает режим «Текст → диаграмма».
Вводит описание и выбирает нотацию.
Система отправляет запрос в AI‑модуль.
AI‑модуль возвращает структурированную модель/DSL.
Система отображает DSL в редакторе и диаграмму — в окне просмотра.
Пользователь корректирует и сохраняет.eraser+1
Альтернативы:
AI‑модуль не справился — система показывает сообщение, предлагает упростить запрос.
3.10. UC10: Генерация диаграммы по изображению
Актор: Пользователь, AI‑CV/LLM‑сервис.
Цель: Оцифровать существующую диорамму (скан, фото, скриншот).
Основной сценарий:
Пользователь выбирает «Изображение → диаграмма».
Загружает изображение.
Система отправляет изображение в модуль распознавания.
Модуль возвращает DSL/структуру.
Система отображает результат, пользователь правит и сохраняет.[todiagram]​
3.11. UC11: Просмотр и анализ ошибок диаграммы
Актор: Пользователь.
Цель: Понять, почему диаграмма некорректна, и устранить проблемы.
Основной сценарий:
Пользователь пытается сгенерировать диаграмму.
Система выявляет синтаксические/семантические ошибки.
Отображает список ошибок с указанием строк и описанием.
Пользователь кликает по ошибке, редактор показывает соответствующее место.
После исправления пользователь повторяет генерацию.
3.12. UC12: Управление настройками пользователя
Актор: Пользователь.
Цель: Настроить поведение и внешний вид системы под себя.
Основной сценарий:
Пользователь открывает «Настройки».
Меняет:
тему (светлая/тёмная);
язык интерфейса;
параметры авто‑рендера;
лимиты по размеру диаграммы (например, предупреждения при превышении).[daily]​
Система сохраняет настройки и применяет их.
